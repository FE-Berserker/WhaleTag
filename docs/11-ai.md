# 11. AI 助手

> Claude Code CLI + Agent SDK 是**可选 AI 组件**(用户安装 `.whaleai` 7z 包,**不**随主安装包内置,见 §12)+ HTTP provider(Ollama / OpenAI 兼容),流式侧栏,工具调用 + 批准,只读护栏。AI IPC 拆成常驻 core(key / CLI 发现 / 组件生命周期)+ 按需 runtime(组件装了才 dynamic-import 注册流式 / inline-edit)。

## 2. Provider 抽象与 IPC 通道

`AiProviderId = 'claude-cli' | 'ollama' | 'openai'`(**3 个字面量**,但 `ollama` 与 `openai` 共享同一个 `HttpChatProvider` runtime)。

`providerFor(id: AiProviderId)` 三元:`'ollama' | 'openai'` → HTTP,其他 → Claude。

**IPC 通道**(常驻 core 在 [`ipc-ai-core.ts`](../src/main/ai/ipc-ai-core.ts);SDK 运行时在 [`ipc-ai-runtime.ts`](../src/main/ai/ipc-ai-runtime.ts),组件装了才注册):

| 通道 | 类型 | 用途 |
|---|---|---|
| `ai:query` | invoke | 发起流式轮次(payload: model / cwd / 历史 / sessionId / mcpServers / attachments) |
| `ai:cancel` | invoke | 中断当前轮次 |
| `ai:prewarm` | invoke | 后台预热 WarmQuery |
| `ai:generateTitle` | invoke | 每会话首轮后自动取标题(HTTP only) |
| `ai:inlineEdit` | invoke | 编辑器选区改写(HTTP only) |
| `ai:setApiKey` / `ai:clearApiKey` / `ai:hasApiKey` | invoke | Anthropic key 管理(safeStorage) |
| `ai:setOpenaiKey` / `ai:clearOpenaiKey` / `ai:hasOpenaiKey` | invoke | OpenAI key 管理(safeStorage) |
| `ai:discoverCli` | invoke | 调用 `findClaudeCliPath` |
| `ai:getComponentState` / `ai:installComponent` / `ai:uninstallComponent` | invoke | 可选 AI 组件状态 + 从 `.whaleai` 安装 / 卸载(见 §12;安装成功后热注册 runtime handler) |
| `ai:resolveApproval` | invoke | 批准弹框结果回传(配 `ai:approvalRequest` push) |

**推送通道**(`event.sender.send`):

- `ai:chunk` —— 流式 chunk(`{ conversationId, chunk }`)
- `ai:error` —— 错误
- `ai:approvalRequest` —— 批准请求(`{ reqId, toolName, input }`),renderer 渲染 `ApprovalModal` + 调 `ai:resolveApproval` 回传

contextIsolation / sandbox 不破。仅固定 `ai:*` 通道,不泛化整桥。

## 3. 推送通道(关键)

Whale 此前**没有任何 main→renderer 推送通道**。AI 流式引入首条:

- 主进程:`event.sender.send('ai:chunk' | 'ai:error' | 'ai:approvalRequest', payload)`
- preload:`onAiChunk(cb)` / `onAiError(cb)` / `onAiApprovalRequest(cb)` 返回 unsubscribe

## 4. 流式与渲染

- **流式粒度**:`includePartialMessages: true`(逐 token `text_delta` / `thinking_delta` / `input_json_delta`)。完整消息也走 `stream_event`。`transformSdkMessage` 用 `TransformState` 状态化,去重 trailing 完整 assistant 消息
- **消息积累**:`streamAccumulator.ts`(纯 reducer,StreamChunk → ChatMessage.contentBlocks)
- **渲染**:`MessageRenderer`(marked + DOMPurify)+ `ToolCall`(折叠卡,支持 subagent 嵌套递归)+ `ThinkingBlock`(折叠)
- **rewind**(对话):`rewindConversation(id, fromMessageId)` 截断到该消息**之前** + `sessionId: null`(否则 Claude resume 重放被截掉的轮次);HTTP 无 session,下次发更短历史。每条消息悬停显 ⏪ 按钮(非流式时)
- **rewind**(文件):HTTP 路径 `write_file` 覆盖前调 `backupRevision` 进 `.whale/revisions/`,用既有 RevisionHistoryDialog 逐文件恢复

## 5. 工具系统(双 provider 共用 `decideToolCall` 闸门)

**Claude 路径**:CLI 自带 Read/Write/Edit/**Bash** 直接碰盘(`cwd` = 当前 location 根 + `additionalDirectories` = 所有 location 根)。`canUseTool` 走 `decideToolCall`。

> **`canUseTool` 的生效前提**:`allowDangerouslySkipPermissions` **只能**在 `'yolo'` 模式开(它启用 `bypassPermissions`)。normal/plan 下若也开,SDK 会强制 `bypassPermissions` → `canUseTool` 被 shadow(永不调用,授权弹窗不弹)→ MCP 工具落进 CLI 的 bypass 路径返回空字段 deny → 校验器抛错。见 [docs/09 §22](./09-known-issues.md)。

**HTTP 路径**(`src/main/ai/providers/http/tools.ts`):

- 工具定义:`read_file` / `list_directory`(读)/ `write_file`(写)
- `READ_TOOLS = {read_file, list_directory}`(HTTP provider 自己的 short-circuit 白名单)
- `WRITE_TOOLS = {write_file}`(HTTP 自己的白名单;**与 `readOnlyGuard.WRITE_TOOLS` 是 2 个不同的 Set,语义不同**)
- 每个执行器经 `assertWithinAllowedRoot` 守护;写经 `atomicWriteText`
- `HttpChatProvider.runTurn` 多轮 agent 循环:OpenAI tool-calling —— 流式 `tool_calls` delta 按 index 累积 `arguments` → READ 工具直接执行 / 非读工具走 `decideToolCall`(调用 `checkReadOnlyGuard` + 批准弹框,与 Claude 共用)→ `role: 'tool'` 回传 → 重新请求;**上限 12 轮** (`MAX_TOOL_ROUNDS = 12`)
- 设置 `aiHttpTools`(默认 `true`);`tools` 字段传 `undefined` → 纯聊天
- **意义**:这是 Whale 哲学的工具模型 —— 工具由 Whale 定义执行 / 写入经批准 + allowedRoots + 原子写 / 只读位置真只读

**两套 `WRITE_TOOLS` Set** 区分:

| 位置 | 内容 | 用途 |
|---|---|---|
| `readOnlyGuard.ts:15-23` | `Write / Edit / MultiEdit / NotebookEdit / write_file` | 批准**前**命中只读根 → 自动拒 |
| `http/tools.ts:47-48` | `write_file` | HTTP provider 内部循环判定"要不要走 decideToolCall" |

`approvalHandler.decideToolCall` 是这两个路径的**统一闸门**;HTTP 调它时它内部调 `checkReadOnlyGuard`,Claude 调它时由 SDK canUseTool 回调接入。

## 6. 只读护栏(`src/main/ai/security/readOnlyGuard.ts`)

- `WRITE_TOOLS = {Write, Edit, MultiEdit, NotebookEdit, write_file}` —— 命中只读根 → 批准前**直接拒**
- `BASH_TOOLS = {Bash, BashOutput, KillShell}` —— **cwd 本身是只读根时拒**(无明确目标路径)
- 纯函数,无 IO;只读位置对 AI 与用户同等硬拒

## 8. Warm Query(消除 Claude CLI 冷启动延迟)

**未走 Claudian 的 MessageChannel 持久流**(~1600 行、并发/路由/重启极易出错),改用 SDK 官方的 `startup()` → `WarmQuery` **预热池**:

- `WarmQuery` 是 one-shot(预热好的进程跑一轮)
- 每轮结束后用下一轮 options 后台 `startup()` 预热
- 下一轮到达若 `optionsKey` 精确匹配就用热进程,**任何异常回落冷 `query()`**(永不比今天差)
- 面板打开 / 切会话也预热(首轮快)

**`optionsKey` 实际字段**(`ClaudeChatRuntime.ts:66-81`):

```ts
JSON.stringify({
  cwd, model, effort, permissionMode,
  cli: pathToClaudeCodeExecutable,
  resume, systemPrompt,
  additionalDirectories: [...].sort(),
  settingSources, mcpServers
})
```

任何一项变化 → 热进程不匹配 → 冷启动。

### 12.1 CLI 路径解析顺序

[src/main/ai/providers/claude/cli/findClaudeCliPath.ts](../src/main/ai/providers/claude/cli/findClaudeCliPath.ts) `findClaudeCLIPath(pathValue?)`:

1. **Settings 覆盖**(`pathValue`)—— 用户在设置里手填的 CLI 路径,最优先
2. **Dev node_modules** —— `bundledCliPath()` 用 `eval('require.resolve')` 绕过 webpack 静态分析;**仅 dev**(`npm run dev`)命中,因为两包是 devDep,packaged build 的 node_modules 里没有 → 返 null
3. **AI 组件** —— `componentCliPath()` 从 `<userData>/components/ai/node_modules/@anthropic-ai/claude-code` 读 `bin/claude(.exe)`(v2.x 单文件可执行;回退 legacy node entrypoint)。**packaged build 的主路径**;优先级高于系统装的 claude,保证 WhaleTag 随组件发的版本是被用的那个
4. **Native install locations** —— Windows:`~/.claude/local/claude.exe` / `AppData/Local/Claude/claude.exe` / `ProgramFiles\Claude\`;POSIX 常见路径
5. **npm-global 包入口** —— `getNpmClaudeCodeEntrypointPaths`
6. **`$PATH`** —— `where` / `which` 枚举

返回 `null` = 没找到(CLI 不可用,Claude provider 走错误提示路径)。

### 12.2 组件包格式与构建

`.whaleai` = 7z 压缩包,内含:

```
manifest.json                         # { component:'ai', version, claudeCodeVersion, sdkVersion, createdAt }
node_modules/
  @anthropic-ai/claude-code/          # 含 bin/claude.exe + 平台 optionalDeps(*-win32-x64 等)
  @anthropic-ai/claude-agent-sdk/
```

[scripts/build-ai-component.js](../scripts/build-ai-component.js) 在**构建机**上 `npm install`(拉 devDep)→ 复制两包 + 它们的 per-platform optionalDeps 到 `release/ai-component-stage/` → 写 manifest → 用 `7zip-bin` 的 7za 压成 `release/components/whaletag-ai-<claudeCodeVersion>.whaleai`。**`npm install` 只装当前平台**,所以每个目标平台要在该平台(或 CI matrix)上各跑一次产平台专用包。当前产物 `whaletag-ai-2.1.198.whaleai` ≈ 159MB。

> **peerDeps 取舍(待实证)**:构建脚本注释称 `@anthropic-ai/claude-agent-sdk` 自包含,declared peerDeps(zod / `@modelcontextprotocol/sdk` / `@anthropic-ai/sdk`)不打进包、`require` 仍成功,所以 `PACKAGES` 只列两包。但 `loadClaudeSdk` 注释又说 peerDeps "在组件 node_modules 里、靠 createRequire 解析"。两者目前未实测对齐 —— 见 [docs/09](./09-known-issues.md) 待办:真机安装流程 smoke 验证 peerDeps 是否缺。

### 12.4 SDK 运行时加载

[src/main/ai/component-resolver.ts](../src/main/ai/component-resolver.ts) `loadClaudeSdk()`(进程内缓存,首次懒加载):

1. **Dev / bundled node_modules** —— `eval('require')('@anthropic-ai/claude-agent-sdk')`(devDep,dev 下命中;packaged 抛 `MODULE_NOT_FOUND` 被吞)
2. **AI 组件** —— `createRequire(<componentDir>/node_modules/__component_self__.js')('@anthropic-ai/claude-agent-sdk')`;anchor 锚在组件 node_modules 内,使 SDK 的 peerDeps 也在该目录解析
3. 都失败 → 抛"未安装 AI 组件,请在 设置 → AI 中安装"

`eval('require')` 藏起来不让 webpack 静态分析(镜像 `findClaudeCliPath` 的 `eval('require.resolve')` 套路);SDK 同时是 main webpack external(见 §15),永不进 bundle。[ClaudeChatRuntime](../src/main/ai/providers/claude/ClaudeChatRuntime.ts) 的 `query()` / `startup()` 都先 `await loadClaudeSdk()` 拿到具名导出再调。

> ✅ **inlineEdit 已走 loadClaudeSdk**(2026-07-12 修):[inlineEdit.ts](../src/main/ai/inlineEdit.ts) 的 Claude 路径改用 `await loadClaudeSdk()`(不再 `import('@anthropic-ai/claude-agent-sdk')`),整段包进 try/catch —— 组件未装 / 冷启失败 → `''`(host 显示 `aiInlineEditClaudeEmpty`),不再 uncaught `MODULE_NOT_FOUND`。与 ClaudeChatRuntime 对齐,packaged build 经组件 node_modules 解析。

### 12.5 IPC 分层注册

  file path 取 revision,故 AI 的 wsd.json 备份**操作者可恢复,不直接进 UI**

  30 天清理但 UI 取不到"取舍)
- **目录目标拒绝**:`stat.isDirectory()` → 报错 `needs a file path`(tag 住空 dir 会产 dangling sidecar entry,有 basename 没文件)
- remove 与 tag 不在 → 返 `No-op`(不视为错误)

### 16.3 AI 对视角与扩展的感知面(2026-07-06 新增)

| 面 | AI 当前能感知到 | 感知途径 |
|---|---|---|
| **当前视角** | `viewMode` + (若 `task`)`subview` + `viewDepth` | `useAiStream.send` 第四参 → `payload.turn` → 主进程 `buildSystemPrompt(ctx)` → 拼 **Perspectives 段** |
| **任务三档数据源** | workflow / quadrant / period 智能标签都是**裸值**(无 `workflow:` / `quadrant:` / `period:` 前缀),sidecar 内只存裸值 | 同上,prompt.ts `Perspectives 段` 显式说明 |
| **扩展能力地图** | 15 个内置扩展(archive-viewer 9 格、pdf-viewer textLayer + 虚拟化、office-viewer 已缓存、text-editor 已合并 text-viewer 等)+ double-iframe 拓扑 + inline-edit 边界 | prompt.ts `Extensions 段`(每轮重发) |
| **inline-edit 边界** | HTTP provider `chat/completions`;Claude 路径走冷启动 `query()`(不接 WarmQuery) | prompt.ts `Extensions 段` 末尾 + `inlineEdit.ts` |
| **`.whale/` 元数据地图** | `wsd.json` 聚合 tag / `wtaglib.json` 标签库 / `index.db` SQLite FTS5 / `revisions/` / `ebook-annotations/` | prompt.ts `Tool-map 段`(Claude 路径可直接 Read / Bash 查) |
| **单选文件内容** | 单文件 + 小文本(≤ 50KB + 白名单扩展)inline 进 `<current_note>` | `useAiStream.send` 第二参 attachment |
| **多选文件路径** | 2+ 选中 → `<selected_files count="N">path1\npath2\n...</selected_files>` 信封,仅路径不读内容 | `useAiStream.send` 第四参 `selectedPaths` → `payload.turn.selectedPaths` |

### 16.4 不立项(显式 out of scope,避免日后重复讨论)

> 旧 §16.3 改名 §16.4;内容不变。
